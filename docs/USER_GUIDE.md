## 系统概述

智能桌面系统是一个基于MCP（Model Context Protocol）架构的智能助手，可以帮助用户完成各种桌面任务，包括文档转换、文件操作、邮件处理、网络请求等。

### 核心特性

- **智能意图识别**：自动理解用户需求
- **任务规划**：基于行为树的智能任务规划系统
- **高效缓存**：基于FAISS向量搜索的混合缓存系统
- **安全沙箱**：文件系统操作安全边界，路径限制、操作白名单、危险操作确认
- **易于扩展**：模块化设计，支持自定义工具
- **行为树执行**：支持复杂任务的灵活执行和条件分支

### 系统架构

```
用户输入
    ↓
意图解析（Intent Parser）
    ↓ 判断意图类型
├─ chat类型 → 直接返回聊天响应
├─ cannot_execute类型 → 返回无法执行的原因
└─ task类型 → 检查缓存
    ├─ 缓存命中（哈希精确匹配或FAISS语义匹配）
    │   └─ 使用缓存的tree_config执行行为树
    └─ 缓存未命中
        └─ LLM生成行为树配置（tree_config）
            └─ 执行行为树（Behavior Tree）
                ↓ 树节点执行
                ├─ 序列节点（Sequence）
                ├─ 选择节点（Selector）
                ├─ 并行节点（Parallel）
                ├─ 动作节点（Action）→ 调用MCP工具
                └─ 条件节点（Condition）→ 基于结果分支
                ↓
            返回结果
```

### 意图识别

系统支持三种意图类型：

1. **task（任务型意图）**
   - 当前可用工具可以完成的操作
   - 用户明确要求执行某个操作（如"转换"、"发送"、"处理"、"生成"、"删除"、"打开"、"查询"等）
   - 例如：文件操作、系统信息、文本处理、网络请求、邮件发送、天气查询、股票查询、新闻查询、快递查询等

2. **chat（聊天型意图）**
   - 当前可用工具无法完成的操作
   - 包括问候、提问、闲聊、咨询等纯对话内容
   - 不执行任何工具
   - 例如："你好"、"早上好"、"什么是人工智能"、"今天天气怎么样"（如果没有天气工具）

3. **cannot_execute（无法执行型意图）**
   - 用户确实想调用工具执行任务，但现有工具或工具的组合步骤不能满足任务执行需要
   - 用户明确要求执行某个操作，但当前工具无法完成
   - 用户提到的任务需要多个步骤，但当前工具无法组合完成
   - 用户提到的功能超出了当前工具的能力范围

### 任务执行流程

系统采用智能缓存策略和行为树执行引擎：

1. **两层缓存匹配**
   - 第一层：哈希精确匹配（基于MD5哈希，响应速度约0.01秒）
   - 第二层：FAISS语义匹配（基于向量相似度，响应速度约0.05秒）

2. **缓存命中流程**
   - 从缓存获取行为树配置（tree_config）
   - 直接使用缓存的tree_config执行行为树
   - 返回执行结果

3. **缓存未命中流程**
   - LLM生成行为树配置（tree_config）
   - 缓存tree_config
   - 执行行为树
   - 返回执行结果

## 行为树系统

行为树是智能桌面系统的核心任务执行引擎，提供了灵活、强大的任务规划和执行能力。

### 什么是行为树？

行为树是一种用于AI和任务规划的树形结构，通过组合不同类型的节点来表示复杂的执行逻辑。

### 行为树节点类型

#### 1. 序列节点（Sequence）
- **功能**：按顺序执行所有子节点，全部成功才返回成功
- **使用场景**：需要按步骤执行的任务
- **示例**：先检查文件，再转换文件，最后发送邮件

#### 2. 选择节点（Selector）
- **功能**：依次执行子节点，直到有一个成功就返回成功
- **使用场景**：尝试多种方法完成任务
- **示例**：尝试PDF转Word，如果失败则尝试其他格式

#### 3. 并行节点（Parallel）
- **功能**：同时执行所有子节点
- **使用场景**：独立任务并行执行
- **示例**：同时转换多个文件

#### 4. 动作节点（Action）
- **功能**：调用MCP工具执行具体操作
- **使用场景**：执行实际的工具操作
- **示例**：调用文档转换工具

#### 5. 条件节点（Condition）
- **功能**：检查条件，根据结果决定执行路径
- **使用场景**：基于运行时结果的动态分支
- **示例**：检查文件是否存在，再决定后续操作

### 行为树执行流程

1. **解析意图**：理解用户需求，确定任务类型
2. **生成配置**：LLM生成行为树配置
3. **构建树**：根据配置构建行为树结构
4. **执行树**：从根节点开始执行，按节点类型的逻辑执行子节点
5. **收集结果**：收集所有节点的执行结果
6. **返回结果**：将执行结果返回给用户

### 行为树的优势

- **灵活性**：支持复杂的条件逻辑和分支
- **可维护性**：模块化设计，易于理解和维护
- **可扩展性**：可以轻松添加新的节点类型和执行逻辑
- **可视化**：支持多种可视化格式，便于调试和分析
- **缓存复用**：支持行为树配置的缓存和复用，提高执行效率

#### 安全沙箱说明

安全沙箱系统为文件系统操作提供安全边界，防止意外操作对系统造成损害。

**路径限制**：
- 允许路径：桌面、文档、下载、图片、音乐、视频、当前工作目录
- 非系统盘支持：允许在 D、E 等非系统盘的任何路径操作
- 禁止路径：Windows、Program Files、System32 等系统目录
- 路径遍历检测：防止使用 `../` 等方式访问沙箱外路径

**操作安全**：
- 操作白名单：只允许安全的操作类型（create、read、write、list、copy、check_permission、search、read_write）
- 危险操作确认：delete、move 操作需要用户确认

### 缓存配置

```yaml
cache:
  enabled: true                  # 是否启用缓存（总开关）
  enable_hash_match: true         # 哈希精确匹配开关
  enable_faiss_match: false       # FAISS语义匹配开关
  type: "faiss"                 # 缓存类型
  cache_dir: "cache"            # 缓存目录
  
  ttl: 604800                   # 缓存过期时间（秒），默认一周
  max_total_size_mb: 1024       # 总大小限制（1GB）
  max_db_size_mb: 768           # SQLite数据库最大768MB
  max_faiss_size_mb: 392       # FAISS索引最大392MB
  max_records: 100000           # 最多100000条记录
  
  cleanup_interval: 3600        # 清理间隔（秒），默认1小时
  cleanup_on_startup: true     # 启动时清理过期缓存
  
  similarity_threshold: 0.85    # 相似度阈值（0-1）
  embedding_model: "nomic-embed-text"  # Embedding模型
```

#### 缓存配置说明

**总开关（enabled）**：
- `true`：启用缓存系统（推荐）
- `false`：完全禁用缓存，每次请求都会重新生成计划

**哈希精确匹配（enable_hash_match）**：
- `true`：启用哈希精确匹配（推荐）
- `false`：禁用哈希精确匹配

**FAISS语义匹配（enable_faiss_match）**：
- `true`：启用FAISS语义匹配
- `false`：禁用FAISS语义匹配（默认）

#### 缓存禁用场景说明

##### 1. 何时禁用FAISS语义匹配（enable_faiss_match: false）

**推荐禁用FAISS语义匹配的场景**：

- **资源受限环境**：系统内存或CPU资源有限，FAISS向量搜索消耗较多资源
- **精确性要求高**：需要确保每次请求都精确匹配，不接受语义相似的结果
- **调试和测试**：开发调试阶段需要精确控制每次请求的行为
- **Embedding模型不可用**：Ollama embedding模型未安装或加载失败
- **缓存数据量小**：缓存记录数量很少，语义匹配意义不大

**影响**：
- 禁用后只能通过哈希精确匹配命中缓存
- 相似但表述不同的请求无法命中缓存
- 缓存命中率降低，但精确性提高
- 减少内存和CPU消耗

##### 2. 何时禁用哈希精确匹配（enable_hash_match: false）

**推荐禁用哈希精确匹配的场景**：

- **仅依赖语义匹配**：希望完全依赖FAISS语义匹配，不使用精确匹配
- **特殊测试场景**：需要测试纯语义匹配的效果
- **避免重复请求**：防止完全相同的请求命中缓存

**影响**：
- 禁用后只能通过FAISS语义匹配命中缓存
- 完全相同的请求也无法命中缓存
- 缓存命中率大幅降低
- 通常不建议单独禁用

##### 3. 何时全局禁用缓存（enabled: false）

**推荐全局禁用缓存的场景**：

- **调试和开发**：需要每次都重新生成计划以验证LLM输出
- **测试新功能**：测试新的意图识别或任务规划逻辑
- **缓存问题排查**：怀疑缓存导致错误结果时临时禁用
- **频繁变化的任务**：任务参数或内容经常变化，缓存效果差
- **存储空间不足**：磁盘空间紧张，无法存储缓存数据
- **性能测试**：需要测试无缓存情况下的系统性能

**影响**：
- 每次请求都会调用LLM生成计划
- 响应时间显著增加
- LLM API调用次数大幅增加
- 不占用磁盘空间

#### 缓存匹配策略

系统采用两层匹配策略：

1. **第一层：哈希精确匹配**
   - 基于请求内容的MD5哈希
   - 完全相同的请求会立即命中
   - 响应速度最快（约0.01秒）

2. **第二层：FAISS语义匹配**
   - 基于向量相似度的语义匹配
   - 相似但表述不同的请求可以命中
   - 响应速度较快（约0.05秒）
   - 受`similarity_threshold`控制，阈值越高命中可能性越低

**TTL（过期时间）**：
- `604800`秒（7天）：默认设置，适合大多数场景
- `3600`秒（1小时）：适合频繁更新的内容
- `86400`秒（24小时）：适合稳定的内容

**大小限制**：
- `max_total_size_mb`：总大小限制，超过后会清理旧缓存
- `max_db_size_mb`：SQLite数据库大小限制，建议设置为总大小的75%
- `max_faiss_size_mb`：FAISS索引大小限制，建议设置为总大小的25%

**相似度阈值**：
- `0.85`：平衡匹配（推荐）
- `0.90`：高精度匹配，缓存命中率较低
- `0.80`：宽松匹配，缓存命中率较高

详细配置指南请参考：[缓存配置指南](./user_config/CACHE_CONFIG_GUIDE.md)

### 执行智能配置

```yaml
execution_intelligence:
  # 工具执行模式配置
  tool_intelligent_mode:
    file_operations: "direct"      # 直接执行
    email_processor: "confirm"    # 需要确认
    weather_query: "infer"        # 自动推断
    system_command: "intelligent" # 智能模式
    text_processing: "direct"
    network_request: "direct"
    system_info: "direct"
    pdf_processor: "direct"
    document_converter: "direct"
```

#### 执行模式说明

**direct（直接执行）**：
- 完全禁用执行智能
- 直接执行工具，不进行推断或确认
- 适用于：低风险、确定性操作

**infer（仅推断）**：
- LLM基于上下文自动推断缺失参数
- 不对用户进行确认
- 适用于：有充分上下文的场景

**confirm（仅确认）**：
- 危险操作前请求用户确认
- 不进行LLM推断
- 适用于：高风险操作（删除文件、发送邮件等）

**intelligent（智能模式）**：
- 结合推断和确认的智能决策
- 高置信度时自动执行，低置信度时请求确认
- 适用于：大多数场景

#### 配置建议

**默认推荐配置**：
- 文件操作：`direct`（基本操作安全，危险操作有沙箱保护）
- 邮件发送：`confirm`（涉及外部通信，需要确认）
- 天气查询：`infer`（参数简单，上下文充分）
- 系统命令：`intelligent`（根据具体情况智能决策）

### 行为树自动修复配置

```yaml
behavior_tree:
  auto_repair:
    enabled: true          # 启用自动修复
    max_repair_attempts: 2 # 最大修复尝试次数
    repair_timeout: 30     # 修复超时时间（秒）
```

#### 自动修复说明

当行为树执行失败时，系统会自动：
1. 分析错误信息
2. 调用LLM分析并修复配置
3. 重新执行修复后的行为树
4. 最多尝试`max_repair_attempts`次

**适用场景**：
- LLM生成的行为树配置有误
- 参数格式不正确
- 节点依赖关系错误

**注意事项**：
- 自动修复会增加响应时间
- 不是所有错误都能自动修复
- 建议在开发调试时启用，生产环境谨慎使用

## 常见问题

### Q1: 意图解析失败怎么办？

**A**: 检查以下几点：
1. 确保LLM服务正常运行
2. 检查LLM配置是否正确
3. 查看日志了解具体错误信息

### Q2: 缓存未命中怎么办？

**A**: 检查以下几点：
1. 确保缓存已启用
2. 检查相似度阈值是否过高
3. 查看日志了解缓存状态

### Q3: 如何提高缓存命中率？

**A**: 可以尝试以下方法：
1. 降低相似度阈值（如从0.85降到0.80）
2. 增加缓存过期时间
3. 使用更相似的表述

### Q4: 如何清理缓存？

**A**: 有以下几种方法：
1. 自动清理：系统会定期清理过期缓存
2. 手动清理：删除cache目录
3. 调整配置：减少TTL时间

### Q5: 系统支持哪些文件格式？

**A**: 支持以下文件格式：
- 文档：PDF、Word（.docx）
- 文本：TXT
- 音频：MP3、WAV（文字转语音）
- 图片：PNG、JPG（OCR识别）

### Q6: 什么是执行智能？如何配置？

**A**: 执行智能是行为树执行阶段的高级功能，支持四种模式：
- **direct**：直接执行
- **infer**：LLM自动推断缺失参数
- **confirm**：危险操作前请求用户确认
- **intelligent**：智能决策（推断+确认）

配置方法：在`user_config/config.yaml`中修改`execution_intelligence.tool_intelligent_mode`。

### Q7: 行为树自动修复是什么？

**A**: 当行为树执行失败时，系统会自动调用LLM分析错误并尝试修复配置，然后重新执行。可以通过`behavior_tree.auto_repair.enabled`启用或禁用。

### Q8: 如何查看行为树执行过程？

**A**: 系统会在日志目录（默认`logs/`）中保存行为树执行记录，包括：
- 行为树配置文件（JSON格式）
- 可视化文件（HTML格式）
- 执行日志

可以使用`visualize_behavior_tree.py`脚本生成可视化图表。
