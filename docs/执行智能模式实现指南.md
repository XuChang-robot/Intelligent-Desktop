# 执行智能模式实现指南

## 概述

执行智能模式是行为树系统中的高级功能，允许工具根据用户配置和上下文智能地决定执行策略。系统通过权限管理和确认机制，为用户提供灵活、安全的工具执行体验。

## 架构设计

### 权限层级

执行智能采用分层权限设计，从最低权限到最高权限：

| 模式 | 权限 | 说明 |
|------|------|------|
| **direct** | 🚫 禁止 | 完全禁用执行智能，直接执行工具 |
| **infer** | 🔵 仅推断 | 只允许 LLM 推断缺失参数，不触发用户确认 |
| **confirm** | 🟡 仅确认 | 只允许触发用户确认，不进行 LLM 推断 |
| **intelligent** | 🟢 全开 | 同时允许 LLM 推断和用户确认，智能决策 |

### 配置流程

```
用户配置 (user_config/config.yaml)
  ↓
  'confirm' | 'infer' | 'intelligent' | 'direct'
  ↓
behavior_tree.py 直接使用用户配置作为 strategy
  ↓
nodes.py 直接将 strategy 作为 execution_mode 传递
  ↓
tool_base.py 处理权限
  ↓
工具根据 execution_mode 决定行为
```

## 工具支持智能模式

### 必需步骤

#### 步骤1：在工具注册函数中添加 execution_mode 参数

在 MCP 工具注册函数中，需要添加 `execution_mode` 和 `ctx` 两个参数：

- `execution_mode`：执行模式参数，接收用户配置的执行策略
- `ctx`：FastMCP 上下文参数，用于触发用户确认

这两个参数都应该设置为可选参数（`Optional`），默认值为 `None`。

#### 步骤2：在 execute 方法中处理 execution_mode

在工具的 `execute` 方法中，根据 `execution_mode` 的值决定是否需要用户确认：

- 如果 `execution_mode` 为 `confirm` 或 `intelligent`，则检查确认权限
- 如果有确认权限，则构建确认消息并触发用户确认
- 如果用户取消，则返回错误结果；否则继续执行

#### 步骤3：调用基类的确认方法

使用基类提供的 `_confirm_with_permission` 方法来触发用户确认：

- 该方法会自动检查权限，只有在有确认权限时才会触发
- 如果权限不足或上下文不存在，则直接返回 `True`（继续执行）
- 如果有权限，则调用 `_trigger_elicitation` 触发用户确认

### 基类提供的接口

#### 权限检查方法

基类提供了以下权限检查方法：

- `get_execution_mode(**kwargs)`：获取执行模式，默认值为 `direct`
- `has_confirm_permission(**kwargs)`：检查是否有确认权限，返回 `True` 或 `False`
- `has_infer_permission(**kwargs)`：检查是否有推断权限，返回 `True` 或 `False`

这些方法会从 `kwargs` 中提取 `execution_mode` 参数，并根据其值返回相应的权限状态。

#### 确认方法

基类提供了 `_confirm_with_permission(ctx, message, **kwargs)` 方法：

- 该方法会自动检查确认权限
- 如果有权限，则触发用户确认
- 返回 `True` 表示继续执行，返回 `False` 表示用户取消

### 配置文件

在 `user_config/config.yaml` 中配置工具的默认执行模式：

```yaml
execution_intelligence:
  # 工具执行模式配置
  tool_intelligent_mode:
    # 不支持智能模式的工具
    simple_tool: "direct"
    
    # 只需要推断的工具
    data_processor: "infer"
    
    # 只需要确认的工具
    email_processor: "confirm"
    file_operations: "confirm"
    
    # 需要智能决策的工具
    system_command: "intelligent"
    network_request: "intelligent"
```

## 实现示例

### 示例1：简单的数据处理工具（不需要智能模式）

**特点**：
- 工具执行低风险操作，不需要用户确认
- 在注册函数中添加 `execution_mode` 参数，但不使用
- 直接执行工具逻辑，不检查权限

**适用场景**：
- 数据格式转换
- 文本处理
- 信息查询

### 示例2：邮件发送工具（需要确认）

**特点**：
- 工具执行高风险操作，需要用户确认
- 在 `execute` 方法中检查确认权限
- 构建详细的确认消息，包含收件人、主题、附件等信息
- 如果用户取消，返回错误结果

**适用场景**：
- 发送邮件
- 发送消息
- 提交表单

### 示例3：文件操作工具（部分操作需要确认）

**特点**：
- 工具执行多种操作，部分需要确认
- 根据操作类型决定是否需要确认
- 删除、移动等破坏性操作需要确认
- 读取、写入等非破坏性操作不需要确认

**适用场景**：
- 文件系统操作
- 数据库操作
- 系统配置修改

## 最佳实践

### 1. 何时需要确认

**需要确认的操作**：
- 删除文件/文件夹
- 移动文件/文件夹
- 发送邮件
- 执行系统命令
- 修改系统配置
- 网络请求（POST/PUT/DELETE）

**不需要确认的操作**：
- 读取文件
- 写入文件（非覆盖）
- 列出目录
- 搜索文件
- 接收邮件
- GET 请求

### 2. 确认消息格式

**好的确认消息**应该包含：
- 操作类型（如"发送邮件"、"删除文件"）
- 关键参数（如收件人、文件路径）
- 附加信息（如附件数量、文件大小）
- 服务器信息（如 SMTP 服务器）

**不好的确认消息**：
- 信息太少，用户无法了解操作详情
- 信息太多，用户难以快速理解
- 格式混乱，难以阅读

### 3. 错误处理

**建议的错误处理模式**：
1. 使用 try-except 捕获异常
2. 记录错误日志，包含异常堆栈
3. 返回友好的错误消息
4. 确保资源正确释放

### 4. 配置建议

**默认模式选择**：
- 高风险操作：使用 `confirm` 模式
- 中等风险操作：使用 `intelligent` 模式
- 低风险操作：使用 `direct` 模式
- 需要智能推断的操作：使用 `infer` 模式

### 常见问题

**问题1：execution_mode 没有传递到工具**

**可能原因**：
- 工具注册函数中没有添加 `execution_mode` 参数
- `safe_execute` 调用中没有传递 `execution_mode`

**解决方法**：
- 检查工具注册函数的参数列表
- 检查 `safe_execute` 调用的参数传递

**问题2：确认没有触发**

**可能原因**：
- `execution_mode` 不是 `confirm` 或 `intelligent`
- 工具中没有正确调用 `_confirm_with_permission`
- `ctx` 参数没有正确传递

**解决方法**：
- 检查 `execution_mode` 的值
- 检查工具中的确认逻辑
- 检查 `ctx` 参数的传递

**问题3：LLM 推断失败**

**可能原因**：
- LLM 服务没有正常运行
- 推断服务配置不正确
- 网络连接问题

**解决方法**：
- 检查 LLM 服务状态
- 检查推断服务配置
- 查看推断服务日志

## 总结

执行智能模式为行为树系统提供了灵活、安全的工具执行机制。工具开发者可以通过以下步骤支持智能模式：

1. 在工具注册函数中添加 `execution_mode` 参数
2. 在 `execute` 方法中检查权限并触发确认
3. 在配置文件中设置工具的默认执行模式
4. 根据操作风险级别决定是否需要确认

通过合理使用执行智能模式，可以提升用户体验，同时确保系统安全性。
