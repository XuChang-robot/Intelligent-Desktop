执行智能的三种策略

概述

在 LLM 与行为树混合架构中，执行阶段的智能分布是一个核心设计问题。当行为树节点面临参数缺失、执行异常或信息不全时，系统需要决定如何引入智能能力来解决问题。本文档详细阐述三种执行智能策略：LLM 推断、Elicitation 用户补全、以及混合模式，并分析其适用场景、实现方式和权衡取舍。

一、策略一：LLM 推断

1.1 核心思想

利用 LLM 的上下文理解能力，基于已有信息自动推断缺失的参数值或决策路径。整个过程对用户透明，不打断执行流程。

1.2 工作流程

节点执行前检查参数
    ↓
发现参数缺失
    ↓
收集相关上下文（黑板数据、用户原始输入、节点定义）
    ↓
调用轻量级 LLM 进行推断
    ↓
解析结构化输出
    ↓
填充缺失参数
    ↓
继续执行节点

1.3 典型应用场景

场景 示例 推断依据
输出路径推断 用户说"转换文件"，推断输出到同目录 输入文件路径、操作习惯
格式选择 根据文件内容推断合适的输出格式 文件扩展名、内容特征
默认值填充 邮件主题默认为"无主题" 参数可选性定义
参数映射 "发给老板"映射到具体邮箱 联系人信息、用户习惯

1.4 实现示例

class LLMInferenceNode(ActionNode):
    """
    LLM 推断节点
    基于上下文自动推断缺失参数
    """

    def __init__(self, inference_config):
        self.missing_params = inference_config['missing_params']
        self.context_keys = inference_config['context_keys']
        self.output_schema = inference_config['output_schema']

    async def execute(self, context):
        # 1. 收集上下文
        local_context = {}
        for key in self.context_keys:
            local_context[key] = context.blackboard.get(key)

        # 2. 构建提示词
        prompt = f"""
        基于以下上下文，推断缺失的参数值：

        用户原始输入：{local_context.get('user_input')}
        可用信息：{local_context}

        需要推断的参数：
        {self.format_missing_params()}

        请输出 JSON 格式：
        {{
            "inferred_params": {{
                "param_name": "inferred_value"
            }},
            "confidence": 0.0-1.0,
            "reasoning": "简要说明推断依据"
        }}
        """

        # 3. 调用轻量级 LLM
        response = await self.llm.call(
            prompt=prompt,
            max_tokens=200,
            temperature=0.1
        )

        # 4. 解析并验证
        inference_result = self.parse_json(response)

        if inference_result['confidence'] < 0.7:
            # 置信度低，需要其他策略补充
            return Status.NEED_CONFIRMATION

        # 5. 填充参数
        for param, value in inference_result['inferred_params'].items():
            context.blackboard.set(f"inferred.{param}", value)

        return Status.SUCCESS

1.5 优势与局限

优势： - ✅ 执行流畅：无感知，不打断用户 - ✅ 响应快速：无需等待用户输入 - ✅ 体验一致：保持自动化执行的流畅感 - ✅ 成本可控：单次轻量 LLM 调用

局限： - ❌ 准确性风险：可能推断错误 - ❌ 意图误解：可能偏离用户真实意图 - ❌ 责任模糊：错误时难以定位责任 - ❌ 适用受限：仅适合有上下文的场景

1.6 适用条件

推荐使用 LLM 推断的条件：

1. 上下文充分性
   - 有足够的信息线索
   - 缺失参数与已知信息有关联

2. 错误成本可控
   - 推断错误不会导致严重后果
   - 可以事后纠正或撤销

3. 用户期望
   - 用户期望快速执行
   - 用户信任系统判断

4. 置信度阈值
   - LLM 对推断结果有较高置信度（>0.7）

二、策略二：Elicitation 用户补全

2.1 核心思想

当关键信息缺失且无法可靠推断时，主动暂停执行流程，通过交互方式引导用户补充所需信息。

2.2 工作流程

节点执行前检查参数
    ↓
发现关键参数缺失
    ↓
判断不可推断（无上下文或后果严重）
    ↓
构建 Elicitation 请求
    ↓
暂停行为树执行
    ↓
向用户展示信息收集界面
    ↓
等待用户响应
    ↓
接收用户输入
    ↓
验证输入有效性
    ↓
恢复执行

2.3 典型应用场景

场景 示例 原因
身份确认 邮件收件人、联系人选择 无法从上下文推断
关键决策 是否覆盖现有文件 涉及用户意图
内容创作 邮件正文、文档内容 需要用户表达
安全确认 删除操作确认 后果不可逆
歧义消解 多个匹配项选择 需要用户指定

2.4 实现示例

class ElicitationNode(ActionNode):
    """
    Elicitation 节点
    引导用户补充关键信息
    """

    def __init__(self, elicitation_config):
        self.required_params = elicitation_config['required_params']
        self.prompt_template = elicitation_config['prompt']
        self.validation_rules = elicitation_config.get('validation', {})

    async def execute(self, context):
        # 1. 检查缺失的关键参数
        missing = self.check_missing_params(context)

        if not missing:
            return Status.SUCCESS

        # 2. 构建 Elicitation 请求
        elicitation_request = {
            'type': 'parameter_elicitation',
            'node_id': self.id,
            'node_name': self.name,
            'missing_params': [
                {
                    'name': param['name'],
                    'description': param['description'],
                    'type': param['type'],
                    'required': param['required'],
                    'examples': param.get('examples', [])
                }
                for param in missing
            ],
            'prompt': self.prompt_template.format(
                node_name=self.name,
                param_list=self.format_param_list(missing)
            ),
            'context': self.get_relevant_context(context)
        }

        # 3. 暂停执行，请求用户输入
        context.suspend_execution(elicitation_request)

        # 4. 等待用户响应（异步）
        user_response = await context.wait_for_user_input(
            timeout=300,  # 5分钟超时
            on_timeout=self.handle_timeout
        )

        # 5. 验证用户输入
        validation_result = self.validate_input(
            user_response, 
            self.validation_rules
        )

        if not validation_result.valid:
            # 验证失败，重新请求
            return Status.RETRY_WITH_FEEDBACK(
                feedback=validation_result.errors
            )

        # 6. 填充参数并恢复
        for param_name, value in user_response.items():
            context.blackboard.set(param_name, value)

        return Status.SUCCESS

    def check_missing_params(self, context):
        """检查缺失的参数"""
        missing = []
        for param in self.required_params:
            value = context.blackboard.get(param['name'])
            if value is None or value == '':
                missing.append(param)
        return missing

2.5 Elicitation 的形式

形式一：阻塞式询问

[系统暂停]

需要您提供以下信息：

📧 收件人邮箱：__________
📝 邮件主题：__________（可选，默认"无主题"）

[确认] [取消]

等待用户输入...

特点： - 执行完全暂停 - 必须等待用户响应 - 适合关键参数

形式二：非阻塞式提示

[系统继续准备其他步骤]

💡 提示：检测到以下信息缺失，将在需要时询问：
   - 收件人邮箱（将在发送前确认）

[立即填写] [稍后处理]

特点： - 不立即阻塞 - 延迟到真正需要时 - 用户体验更流畅

形式三：渐进式收集

步骤 1/3：基本信息
📧 收件人：__________

[下一步]

步骤 2/3：邮件内容
📝 正文：__________

[下一步] [上一步]

步骤 3/3：确认发送
预览邮件内容...

[发送] [修改] [取消]

特点： - 分步骤收集 - 每步验证 - 适合复杂信息

2.6 优势与局限

优势： - ✅ 信息准确：直接来自用户 - ✅ 用户控制：用户掌握决策权 - ✅ 意图明确：避免误解 - ✅ 责任清晰：用户对自己的输入负责

局限： - ❌ 打断体验：破坏自动化流程 - ❌ 等待成本：需要用户响应时间 - ❌ 流程复杂：需要处理超时、取消等 - ❌ 不适合批处理：无法自动化执行

2.7 适用条件

推荐使用 Elicitation 的条件：

1. 关键性
   - 参数对任务成功至关重要
   - 缺失会导致任务失败

2. 不可推断性
   - 无法从上下文推断
   - 推断可靠性低

3. 后果严重性
   - 错误代价高
   - 难以撤销或纠正

4. 用户意图核心
   - 涉及用户的主观意愿
   - 需要用户表达偏好

5. 歧义性
   - 存在多个可能选项
   - 需要用户明确指定

三、策略三：混合模式（推断+确认）

3.1 核心思想

结合 LLM 推断的效率和 Elicitation 的准确性：先由 LLM 推断缺失参数，再向用户展示推断结果请求确认，在效率和准确性之间取得平衡。

3.2 工作流程

节点执行前检查参数
    ↓
发现参数缺失
    ↓
收集上下文
    ↓
LLM 推断缺失参数
    ↓
评估推断置信度
    ↓
高置信度 ──→ 自动执行（可配置）
    ↓
中置信度 ──→ 展示确认
    ↓
低置信度 ──→ 触发 Elicitation
    ↓
用户确认/修改
    ↓
继续执行

3.3 典型应用场景

场景 推断内容 确认方式
文件转换 输出格式、路径 展示推断结果，5秒倒计时
邮件发送 主题、收件人 展示完整邮件预览
数据导出 导出范围、格式 展示参数摘要
批量操作 操作范围 展示影响文件列表

3.4 实现示例

class HybridIntelligenceNode(ActionNode):
    """
    混合智能节点
    推断 + 确认
    """

    def __init__(self, config):
        self.inference_config = config['inference']
        self.confirmation_config = config['confirmation']
        self.elicitation_config = config['elicitation']

        # 置信度阈值
        self.auto_execute_threshold = 0.85  # 自动执行
        self.confirm_threshold = 0.60       # 需要确认
        # < 0.60 触发 Elicitation

    async def execute(self, context):
        # 1. 检查缺失参数
        missing = self.check_missing_params(context)

        if not missing:
            return await self.execute_action(context)

        # 2. LLM 推断
        inference_result = await self.infer_missing_params(
            missing, context
        )

        confidence = inference_result['confidence']
        inferred_params = inference_result['inferred_params']

        # 3. 根据置信度选择策略
        if confidence >= self.auto_execute_threshold:
            # 高置信度：自动执行（可配置）
            if self.confirmation_config.get('silent_auto_execute'):
                return await self.execute_with_inferred(
                    inferred_params, context
                )
            else:
                # 即使高置信度也简要通知
                return await self.execute_with_notification(
                    inferred_params, context
                )

        elif confidence >= self.confirm_threshold:
            # 中置信度：请求确认
            return await self.request_confirmation(
                inferred_params, 
                inference_result['reasoning'],
                context
            )

        else:
            # 低置信度：触发 Elicitation
            return await self.trigger_elicitation(
                missing, 
                inferred_params,  # 作为默认值
                context
            )

    async def request_confirmation(self, inferred_params, reasoning, context):
        """请求用户确认推断结果"""

        confirmation_request = {
            'type': 'confirmation',
            'title': '请确认以下操作',
            'content': self.format_confirmation_content(
                inferred_params, reasoning
            ),
            'options': [
                {'label': '确认执行', 'value': 'confirm', 'primary': True},
                {'label': '修改参数', 'value': 'modify'},
                {'label': '取消', 'value': 'cancel'}
            ],
            'countdown': self.confirmation_config.get('countdown_seconds', 5),
            'default_action': 'confirm'  # 倒计时结束后默认执行
        }

        # 暂停执行，等待确认
        context.suspend_execution(confirmation_request)

        user_choice = await context.wait_for_user_choice(
            timeout=self.confirmation_config.get('timeout', 30)
        )

        if user_choice == 'confirm' or user_choice == 'timeout':
            # 确认或超时（默认确认）
            return await self.execute_with_inferred(
                inferred_params, context
            )

        elif user_choice == 'modify':
            # 用户要修改，触发带默认值的 Elicitation
            return await self.trigger_elicitation_with_defaults(
                inferred_params, context
            )

        else:  # cancel
            return Status.CANCELLED

    def format_confirmation_content(self, params, reasoning):
        """格式化确认内容"""
        content = "系统将基于以下推断执行操作：\n\n"

        for param_name, value in params.items():
            content += f"• {self.get_param_label(param_name)}: {value}\n"

        content += f"\n推断依据：{reasoning}\n"
        content += f"\n{self.confirmation_config.get('countdown_seconds', 5)}秒后自动执行..."

        return content

3.6 优势与局限

优势： - ✅ 平衡效率与准确：高置信度自动执行，低置信度请求确认 - ✅ 用户可控：关键决策由用户把关 - ✅ 渐进式信任：随着系统了解用户习惯，自动执行比例提升 - ✅ 容错性强：多种降级路径

局限： - ❌ 实现复杂：需要处理多种分支 - ❌ 配置繁琐：需要设定多个阈值和策略 - ❌ 用户学习成本：需要理解系统的确认机制

3.7 适用条件

推荐使用混合模式的条件：

1. 参数重要性中等
   - 不是关键参数，但也不是完全可选
   - 错误有一定影响，但可接受

2. 可推断但有不确定性
   - 有上下文线索，但不够充分
   - 推断置信度中等（60%-85%）

3. 用户期望平衡
   - 既希望自动化，又希望保持控制
   - 愿意在关键决策上花时间确认

4. 场景多样性
   - 不同场景需要不同策略
   - 无法单一策略覆盖

4.3 选择指南

选择 LLM 推断： - 参数重要性低 - 上下文充分 - 推断可靠性高 - 用户期望快速执行

选择 Elicitation： - 参数重要性高 - 无法推断或推断不可靠 - 涉及用户主观意图 - 后果严重

选择混合模式： - 参数重要性中等 - 推断有一定可靠性但不完全确定 - 需要平衡效率和控制 - 场景多样

六、潜在优化空间

6.1 置信度阈值的动态调整

当前设计使用固定阈值（0.85/0.60），但实际应用中，不同场景的最佳阈值可能不同。建议引入动态调整机制：

threshold_adaptation:
  # 不同类型节点使用不同阈值
  per_node_type: true
  node_thresholds:
    file_operation:
      auto_execute: 0.80
      confirm: 0.55
    email_operation:
      auto_execute: 0.90
      confirm: 0.70
    delete_operation:
      auto_execute: 0.95
      confirm: 0.80

  # 基于历史数据动态调整
  learning_enabled: true
  learning_rate: 0.1
  adjustment_window: 100  # 基于最近100次决策调整

  # 用户偏好设置
  user_preference:
    options: ["prefer_auto", "prefer_confirm", "prefer_ask"]
    default: "prefer_confirm"
    impact:
      prefer_auto: [-0.10, -0.10]  # 降低阈值，更多自动执行
      prefer_confirm: [0, 0]        # 使用默认阈值
      prefer_ask: [0.10, 0.10]      # 提高阈值，更多询问

  # 根据上下文复杂程度调整
  context_sensitive: true
  complexity_factors:
    - param_count        # 参数数量
    - data_ambiguity     # 数据歧义度
    - user_history_match # 用户历史匹配度

动态调整算法示例：

class AdaptiveThresholdManager:
    """自适应阈值管理器"""

    def __init__(self):
        self.base_thresholds = {
            'auto_execute': 0.85,
            'confirm': 0.60
        }
        self.adjustment_history = []

    def get_adjusted_threshold(self, node_type, context, user_preference):
        """获取调整后的阈值"""
        thresholds = self.base_thresholds.copy()

        # 1. 节点类型调整
        if node_type in self.node_thresholds:
            thresholds = self.node_thresholds[node_type].copy()

        # 2. 用户偏好调整
        pref_adjustment = self.get_preference_adjustment(user_preference)
        thresholds['auto_execute'] += pref_adjustment[0]
        thresholds['confirm'] += pref_adjustment[1]

        # 3. 上下文复杂度调整
        complexity = self.calculate_context_complexity(context)
        complexity_adjustment = complexity * 0.05  # 复杂度影响
        thresholds['auto_execute'] -= complexity_adjustment
        thresholds['confirm'] -= complexity_adjustment

        # 4. 历史学习调整
        if self.learning_enabled:
            history_adjustment = self.learn_from_history(node_type)
            thresholds['auto_execute'] += history_adjustment
            thresholds['confirm'] += history_adjustment

        return thresholds

    def calculate_context_complexity(self, context):
        """计算上下文复杂度"""
        factors = []

        # 参数数量
        param_count = len(context.get('missing_params', []))
        factors.append(min(param_count / 5, 1.0))  # 归一化

        # 数据歧义度
        ambiguity = context.get('data_ambiguity', 0)
        factors.append(ambiguity)

        # 用户历史匹配度
        history_match = context.get('user_history_match', 0.5)
        factors.append(1 - history_match)  # 匹配度低则复杂度高

        return sum(factors) / len(factors)

        elif inference['source'] == 'user_history':
            return 'HISTORY_MISMATCH'        # 历史不匹配

class InferenceLearningSystem:
    """推断学习系统"""

    def __init__(self):
        self.correction_history = []
        self.user_patterns = {}

    def record_correction(self, inference, user_correction):
        """记录用户修正"""
        record = {
            'timestamp': datetime.now(),
            'node_type': inference['node_type'],
            'param_name': inference['param_name'],
            'inferred_value': inference['value'],
            'user_value': user_correction['value'],
            'context': inference['context'],
            'was_different': inference['value'] != user_correction['value']
        }

        self.correction_history.append(record)

        if record['was_different']:
            self.update_patterns(record)

    def update_patterns(self, record):
        """更新用户模式"""
        key = f"{record['node_type']}.{record['param_name']}"

        if key not in self.user_patterns:
            self.user_patterns[key] = {
                'corrections': [],
                'preference_score': {}
            }

        # 记录修正
        self.user_patterns[key]['corrections'].append({
            'from': record['inferred_value'],
            'to': record['user_value'],
            'context': record['context']
        })

        # 更新偏好分数
        self.user_patterns[key]['preference_score'][record['user_value']] = \
            self.user_patterns[key]['preference_score'].get(record['user_value'], 0) + 1

    def get_learned_preference(self, node_type, param_name, context):
        """获取学习到的偏好"""
        key = f"{node_type}.{param_name}"

        if key not in self.user_patterns:
            return None

        patterns = self.user_patterns[key]

        # 基于相似上下文找到最可能的偏好
        best_match = self.find_best_context_match(
            patterns['corrections'], context
        )

        if best_match:
            return {
                'value': best_match['to'],
                'confidence': self.calculate_pattern_confidence(patterns),
                'source': 'learned_preference'
            }

        return None

class InferenceCostMonitor:
    """推断成本监控"""

    def __init__(self):
        self.session_costs = {
            'total_tokens': 0,
            'total_calls': 0,
            'by_model': {},
            'by_node_type': {}
        }

    def track_inference(self, model, tokens, node_type, success):
        """追踪推断调用"""
        self.session_costs['total_tokens'] += tokens
        self.session_costs['total_calls'] += 1

        # 按模型统计
        if model not in self.session_costs['by_model']:
            self.session_costs['by_model'][model] = {
                'calls': 0, 'tokens': 0, 'successes': 0
            }
        self.session_costs['by_model'][model]['calls'] += 1
        self.session_costs['by_model'][model]['tokens'] += tokens
        if success:
            self.session_costs['by_model'][model]['successes'] += 1

        # 按节点类型统计
        if node_type not in self.session_costs['by_node_type']:
            self.session_costs['by_node_type'][node_type] = {
                'calls': 0, 'tokens': 0
            }
        self.session_costs['by_node_type'][node_type]['calls'] += 1
        self.session_costs['by_node_type'][node_type]['tokens'] += tokens

    def check_budget(self, task_id, estimated_tokens):
        """检查预算"""
        current_usage = self.session_costs['total_tokens']
        budget = self.get_task_budget(task_id)

        if current_usage + estimated_tokens > budget:
            return {
                'allowed': False,
                'reason': 'budget_exceeded',
                'current': current_usage,
                'budget': budget,
                'remaining': budget - current_usage
            }

        return {'allowed': True}

6.5 失败处理链的完整性

当前设计主要针对"信息缺失"场景，对"推断失败"的处理需要更完善的回退链：

推断失败处理链：

LLM 推断失败
    │
    ├─ Step 1: 重试机制
    │   ├─ 重试 1 次（相同参数）
    │   └─ 重试 1 次（不同模型/参数）
    │       └─ 成功 → 继续
    │       └─ 失败 → Step 2
    │
    ├─ Step 2: 降级推断
    │   ├─ 规则推断（如果有预设规则）
    │   │   └─ 成功 → 继续
    │   │   └─ 失败 → Step 3
    │   └─ 默认值填充（如果参数可选）
    │       └─ 成功 → 继续
    │       └─ 失败 → Step 3
    │
    ├─ Step 3: 轻量级 Elicitation
    │   ├─ 只询问最关键参数
    │   ├─ 其他参数使用默认值
    │   └─ 成功 → 继续
    │       └─ 失败 → Step 4
    │
    └─ Step 4: 最终处理
        ├─ 记录失败详情
        ├─ 通知用户
        └─ 任务降级处理
            ├─ 跳过当前节点
            ├─ 使用替代方案
            └─ 完全取消任务

失败处理链实现：

class InferenceFailureHandler:
    """推断失败处理器"""

    async def handle_failure(self, error, context):
        """处理推断失败"""

        # Step 1: 重试机制
        retry_result = await self.retry_with_fallback(error, context)
        if retry_result.success:
            return retry_result

        # Step 2: 降级推断
        fallback_result = await self.fallback_inference(error, context)
        if fallback_result.success:
            return fallback_result

        # Step 3: 轻量级 Elicitation
        elicitation_result = await self.lightweight_elicitation(error, context)
        if elicitation_result.success:
            return elicitation_result

        # Step 4: 最终处理
        return await self.final_handler(error, context)

    async def retry_with_fallback(self, error, context):
        """重试机制"""
        # 第一次重试：相同参数
        result = await self.retry_inference(context)
        if result.success:
            return result

        # 第二次重试：切换模型
        result = await self.retry_with_different_model(context)
        if result.success:
            return result

        return FailureResult("重试失败")

    async def fallback_inference(self, error, context):
        """降级推断"""
        # 尝试规则推断
        rule_result = self.apply_rules(context)
        if rule_result:
            return SuccessResult(rule_result, source='rule')

        # 尝试默认值
        default_result = self.apply_defaults(context)
        if default_result:
            return SuccessResult(default_result, source='default')

        return FailureResult("降级推断失败")

    async def lightweight_elicitation(self, error, context):
        """轻量级 Elicitation"""
        # 只询问关键参数
        critical_params = self.identify_critical_params(context)

        if not critical_params:
            return FailureResult("无关键参数需要询问")

        # 简化的询问界面
        result = await self.quick_ask(critical_params, context)
        return result

    async def final_handler(self, error, context):
        """最终处理"""
        # 记录失败
        self.log_failure(error, context)

        # 根据节点类型决定处理方式
        node_type = context.get('node_type')

        if self.is_skipable(node_type):
            return SkipResult("跳过节点")
        elif self.has_alternative(node_type):
            alt_result = await self.execute_alternative(context)
            return alt_result
        else:
            return CancelResult("任务取消", reason=str(error))

        for node_id, history in self.inference_history.items():
            if node_id.startswith(node_type):
                for record in history:
                    if record['user_correction']:
                        patterns.append({
                            'inferred': record['inferred_params'],
                            'corrected': record['user_correction'],
                            'context': record.get('context')
                        })

        return patterns

7.3 与日志监控系统的集成

记录每次推断的详细信息，用于后续分析和优化：

class InferenceLogger:
    """推断日志记录器"""

    def __init__(self, log_path):
        self.log_path = log_path
        self.session_logs = []

    def log_inference(self, log_entry):
        """记录推断日志"""
        full_log = {
            'timestamp': datetime.now().isoformat(),
            'session_id': log_entry.get('session_id'),
            'task_id': log_entry.get('task_id'),

            # 节点信息
            'node_id': log_entry['node_id'],
            'node_type': log_entry['node_type'],
            'node_name': log_entry['node_name'],

            # 参数信息
            'missing_params': log_entry['missing_params'],
            'available_context': log_entry.get('available_context', {}),

            # 推断结果
            'inference_result': log_entry['inference_result'],
            'confidence': log_entry['confidence'],
            'strategy_used': log_entry['strategy'],  # inference/elicitation/hybrid

            # 用户交互
            'user_action': log_entry.get('user_action'),  # confirmed/modified/cancelled
            'user_modification': log_entry.get('user_modification'),

            # 性能指标
            'latency_ms': log_entry['latency_ms'],
            'tokens_used': log_entry.get('tokens_used', 0),
            'model_used': log_entry.get('model_used'),

            # 结果
            'success': log_entry.get('success'),
            'error': log_entry.get('error')
        }

        self.session_logs.append(full_log)
        self.persist_log(full_log)

    def persist_log(self, log_entry):
        """持久化日志"""
        log_file = os.path.join(
            self.log_path, 
            f"inference_{datetime.now().strftime('%Y%m%d')}.jsonl"
        )

        with open(log_file, 'a', encoding='utf-8') as f:
            f.write(json.dumps(log_entry, ensure_ascii=False) + '\n')

    def get_statistics(self, time_range=None):
        """获取统计信息"""
        logs = self.filter_logs(time_range)

        return {
            'total_inferences': len(logs),
            'by_strategy': self.group_by(logs, 'strategy_used'),
            'by_node_type': self.group_by(logs, 'node_type'),
            'avg_confidence': self.calculate_avg(logs, 'confidence'),
            'avg_latency_ms': self.calculate_avg(logs, 'latency_ms'),
            'total_tokens': sum(log.get('tokens_used', 0) for log in logs),
            'user_modification_rate': self.calculate_rate(
                logs, lambda x: x.get('user_action') == 'modified'
            ),
            'success_rate': self.calculate_rate(
                logs, lambda x: x.get('success') == True
            )
        }

八、总结

执行智能的三种策略代表了自动化与可控性之间的不同权衡：

策略 核心哲学 最佳场景
LLM 推断 效率优先 高可靠推断、低错误成本
Elicitation 准确优先 关键参数、用户意图核心
混合模式 平衡之道 中等重要性、不确定场景

在实际系统设计中，建议：

  1. 从简单开始：先实现 LLM 推断，覆盖简单场景
  2. 逐步增强：添加 Elicitation 处理关键参数
  3. 智能融合：最终采用混合模式，动态选择策略
  4. 数据驱动：基于实际使用数据持续优化

三种策略并非互斥，而是构成了执行智能的连续谱系。优秀的系统设计应该能够根据场景特征动态选择或组合使用这些策略，在效率、准确性和用户体验之间找到最佳平衡点。
