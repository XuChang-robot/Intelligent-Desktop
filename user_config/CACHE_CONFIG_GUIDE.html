缓存配置指南

概述

智能桌面系统采用基于FAISS向量搜索的混合缓存系统，支持两层匹配策略： 1. 哈希精确匹配：基于请求内容的MD5哈希进行精确匹配 2. FAISS语义匹配：基于向量相似度进行语义匹配

此外，系统还支持行为树缓存，用于缓存常用任务的行为树结构，提高执行速度。

本指南将帮助您根据实际使用场景配置缓存系统。

缓存配置项

基本配置

cache:
  enabled: true                  # 是否启用缓存（总开关）
  enable_hash_match: true         # 哈希精确匹配开关
  enable_faiss_match: false       # FAISS语义匹配开关
  type: "faiss"                 # 缓存类型
  cache_dir: "cache"            # 缓存目录

时间和大小配置

cache:
  ttl: 604800                   # 缓存过期时间（秒），默认一周
  max_total_size_mb: 1024       # 总大小限制（1GB）
  max_db_size_mb: 768           # SQLite数据库最大768MB
  max_faiss_size_mb: 392       # FAISS索引最大392MB
  max_records: 100000           # 最多100000条记录

清理策略配置

cache:
  cleanup_interval: 3600        # 清理间隔（秒），默认1小时
  cleanup_on_startup: true     # 启动时清理过期缓存

匹配参数配置

cache:
  similarity_threshold: 0.85    # 相似度阈值（0-1）
  embedding_model: "nomic-embed-text"  # Embedding模型

缓存禁用场景说明

1. 何时禁用FAISS语义匹配（enable_faiss_match: false）

推荐禁用FAISS语义匹配的场景：

  • 资源受限环境：系统内存或CPU资源有限，FAISS向量搜索消耗较多资源
  • 精确性要求高：需要确保每次请求都精确匹配，不接受语义相似的结果
  • 调试和测试：开发调试阶段需要精确控制每次请求的行为
  • Embedding模型不可用：Ollama embedding模型未安装或加载失败
  • 缓存数据量小：缓存记录数量很少，语义匹配意义不大

影响： - 禁用后只能通过哈希精确匹配命中缓存 - 相似但表述不同的请求无法命中缓存 - 缓存命中率降低，但精确性提高 - 减少内存和CPU消耗

配置示例：

cache:
  enabled: true
  enable_hash_match: true
  enable_faiss_match: false  # 禁用FAISS语义匹配

2. 何时禁用哈希精确匹配（enable_hash_match: false）

推荐禁用哈希精确匹配的场景：

  • 仅依赖语义匹配：希望完全依赖FAISS语义匹配，不使用精确匹配
  • 特殊测试场景：需要测试纯语义匹配的效果
  • 避免重复请求：防止完全相同的请求命中缓存

影响： - 禁用后只能通过FAISS语义匹配命中缓存 - 完全相同的请求也无法命中缓存 - 缓存命中率大幅降低 - 通常不建议单独禁用

配置示例：

cache:
  enabled: true
  enable_hash_match: false  # 禁用哈希精确匹配
  enable_faiss_match: true

3. 何时全局禁用缓存（enabled: false）

推荐全局禁用缓存的场景：

  • 调试和开发：需要每次都重新生成计划以验证LLM输出
  • 测试新功能：测试新的意图识别或任务规划逻辑
  • 缓存问题排查：怀疑缓存导致错误结果时临时禁用
  • 频繁变化的任务：任务参数或内容经常变化，缓存效果差
  • 存储空间不足：磁盘空间紧张，无法存储缓存数据
  • 性能测试：需要测试无缓存情况下的系统性能

影响： - 每次请求都会调用LLM生成计划 - 响应时间显著增加 - LLM API调用次数大幅增加 - 不占用磁盘空间

配置示例：

cache:
  enabled: false  # 全局禁用缓存

缓存匹配策略

系统采用两层匹配策略：

  1. 第一层：哈希精确匹配
  2. 基于请求内容的MD5哈希
  3. 完全相同的请求会立即命中

  4. 响应速度最快（约0.01秒）

  5. 第二层：FAISS语义匹配

  6. 基于向量相似度的语义匹配
  7. 相似但表述不同的请求可以命中
  8. 响应速度较快（约0.05秒）
  9. 受similarity_threshold控制

存储分析

实际数据比例

  • SQLite每条记录：4.61 KB
  • FAISS每条记录：3.00 KB
  • SQLite/FAISS比例：1.54:1

考虑索引和查询开销

  • SQLite需要额外的索引空间（timestamp, intent_hash, faiss_id, last_accessed）
  • SQLite查询需要更多的内存和CPU资源
  • 建议配置比例：4:1

推荐配置

方案1：保守配置（推荐）

cache:
  # 总大小限制
  max_total_size_mb: 1024  # 1GB

  # 大小策略（4:1比例）
  max_db_size_mb: 819      # SQLite最大819MB（80%）
  max_faiss_size_mb: 204  # FAISS最大204MB（20%）

  # 记录数量
  max_records: 10000       # 最多10000条记录

  # 时间策略
  ttl: 604800              # 缓存过期时间（秒），默认一周

  # 清理策略
  cleanup_interval: 3600   # 清理间隔（秒），默认1小时
  cleanup_on_startup: true # 启动时清理过期缓存

适用场景： - 系统资源充足 - 需要高缓存命中率 - SQLite查询频繁

方案2：平衡配置

cache:
  # 总大小限制
  max_total_size_mb: 1024  # 1GB

  # 大小策略（3:1比例）
  max_db_size_mb: 768      # SQLite最大768MB（75%）
  max_faiss_size_mb: 256  # FAISS最大256MB（25%）

  # 记录数量
  max_records: 10000       # 最多10000条记录

  # 时间策略
  ttl: 604800              # 缓存过期时间（秒），默认一周

  # 清理策略
  cleanup_interval: 3600   # 清理间隔（秒），默认1小时
  cleanup_on_startup: true # 启动时清理过期缓存

适用场景： - 系统资源适中 - 平衡缓存命中率和存储效率 - 一般使用场景

方案3：激进配置

cache:
  # 总大小限制
  max_total_size_mb: 1024  # 1GB

  # 大小策略（7:1比例）
  max_db_size_mb: 900      # SQLite最大900MB（88%）
  max_faiss_size_mb: 124  # FAISS最大124MB（12%）

  # 记录数量
  max_records: 10000       # 最多10000条记录

  # 时间策略
  ttl: 604800              # 缓存过期时间（秒），默认一周

  # 清理策略
  cleanup_interval: 3600   # 清理间隔（秒），默认1小时
  cleanup_on_startup: true # 启动时清理过期缓存

适用场景： - SQLite查询非常频繁 - FAISS查询相对较少 - 系统资源充足

清理策略建议

TTL（过期时间）

  • 建议设置：604800秒（7天）
  • 原因：7天后缓存可能不再有效，清理可以节省空间

清理间隔

  • 建议设置：3600秒（1小时）
  • 原因：每小时清理一次过期缓存，避免堆积

启动时清理

  • 建议设置：true
  • 原因：启动时清理过期缓存，确保系统启动后缓存是干净的

配置选择建议

根据使用场景选择

  1. 高频查询场景：选择方案1（保守配置）
  2. 一般使用场景：选择方案2（平衡配置）
  3. SQLite为主场景：选择方案3（激进配置）

根据系统资源选择

  1. 资源充足：选择方案1或方案3
  2. 资源适中：选择方案2
  3. 资源受限：降低max_total_size_mb和max_records

注意事项

  1. SQLite大小设置过小可能导致缓存频繁清理
  2. FAISS大小设置过小可能导致向量搜索失败
  3. TTL设置过短会降低缓存命中率
  4. 清理间隔设置过长会导致缓存堆积
  5. 建议定期监控缓存使用情况，根据实际情况调整配置
